# 宠物心灵翻译器 - 产品需求文档 (PRD)

## 1. 产品概述与愿景

- **产品名称**：宠物心灵翻译器（暂定）
- **一句话定义**：一个面向宠物主人的在线网站（手机端优先），通过录入猫或狗的叫声，利用大模型分析其情绪并转化为人类语音输出，让主人“听懂”宠物在说什么。
- **愿景**：消除人与宠物之间的沟通壁垒，帮助宠物家庭建立更深入的情感连接，让每位主人都能即时读懂宠物的内心世界。
- **核心价值**：将宠物叫声中的情绪（如饥饿、害怕、开心、撒娇等）转化为具体的人话描述，并通过语音合成让主人听见“宠物在说话”。

## 2. 目标用户与使用场景

### 2.1 目标用户

- **主要用户**：养猫或养狗的宠物主人（年龄 20–45 岁，习惯使用智能手机）
- **次要用户**：宠物店员工、兽医、宠物行为训练师等专业人士

### 2.2 使用场景

| 场景 | 说明 |
|------|------|
| 日常情绪解读 | 主人发现宠物发出异常叫声，想立刻知道它想表达什么（害怕、疼痛、饥饿等） |
| 历史行为复盘 | 主人上传之前录制的音频文件，分析特定事件下的情绪变化 |
| 趣味互动 | 主人纯粹想听宠物“说话”，用于社交分享或娱乐 |
| 专业辅助 | 兽医或训练师记录宠物叫声，辅助诊断或行为分析 |

- **输入方式**：支持实时录音 & 上传已有的音频文件（wav / mp3 / m4a 等常见格式）
- **输出**：识别出的情绪 + 一段自然语言描述 + 语音合成的人话（TTS）
- **语言**：中文 / 英文（用户可选择输出语言）

## 3. 核心问题与解决方案

### 3.1 核心问题

1. 宠物主人无法准确理解猫狗叫声背后的真实情绪与需求（饥饿、害怕、求助、开心等）
2. 传统宠物叫声分析需要专业知识，普通用户无工具可用
3. 市场上缺乏将分析结果直接转换为“宠物说话”的语音交互产品
4. 移动端使用体验差，操作繁琐

### 3.2 解决方案

- **录音 + 上传双通道**：满足即时性和复盘两种需求
- **大模型情绪分析**：调用第三方语音识别 API 提取特征，再由大模型推理出情绪含义
- **TTS 语音合成**：将分析结果用拟人化口吻（例如：“妈妈我饿了！”）朗读出来，增强趣味性和共情体验
- **手机端优先响应式设计**：基于 Next.js 实现移动友好界面，录音功能适配浏览器 MediaRecorder API

## 5. 非功能性需求

### 5.1 性能

- **分析响应时间**：从录音结束（或上传完成）到语音合成播放，应在 **10 秒内** 完成（用户可接受 3–10 秒，避免超过 20 秒）
- **并发支持**：需承受 **日活 1000 次浏览**，高峰时段同时分析请求数预估 50–100 QPS（基于 TikTok 广告效应）。Vercel 自动弹性扩展 + Supabase 连接池上限建议 200
- **文件上传**：单文件上传耗时 < 5 秒（20MB 以内），使用 Supabase Storage 的 CDN 加速

### 5.2 可靠性

- **错误处理**：第三方 API 调用失败时，前端展示友好错误提示并允许重试；超过 3 次重试采用本地缓存模拟结果（显示“分析暂时不可用”）
- **数据备份**：Supabase 每天自动备份用户记录（PostgreSQL 日常维护）

### 5.3 安全与合规

- **数据加密**：所有用户音频文件存储时加密（Supabase Storage 默认 Server-side 加密）；传输使用 HTTPS
- **隐私合规**：符合 **GDPR / CCPA** 基本要求（用户可删除自己的账号及所有数据；提供隐私政策页面；不收集非必要个人信息）
- **音频保留**：默认保留 90 天，用户可在设置中手动清除

### 5.4 可用性

- **移动端适配**：页面宽度 320–480px 完美显示，按钮至少 44x44pt
- **离线兼容性**：录音功能在无网络时可暂存本地，有网后自动上传（P1 需求）
- **无障碍**：按键有 aria-label，录音状态有声音+视觉反馈

## 7. 风险与假设

### 7.1 主要风险

| 风险 | 影响 | 可能性 | 应对措施 |
|------|------|--------|----------|
| 第三方语音识别 API 对猫狗叫声准确率低 | 分析结果不可用，用户流失 | 高 | 选择支持动物声音的专用 API（如 ElevenLabs 动物识别）；结果中加入置信度提示 |
| TikTok 广告带来瞬时高并发，Vercel 冷启动延迟 | 首页加载/首次录音分析超时 | 中 | 使用 Vercel 的 Pro 计划（预留并发实例）；开启 Edge Functions 缓存静态资源 |
| TTS 合成延迟或费用超预算 | 用户体验差，成本失控 | 中 | 预留缓存机制（相同结果不重复合成）；设置每日调用上限告警 |
| 用户录音隐私泄露 | 法律诉讼与品牌危机 | 低 | 实施严格 GDPR 合规，所有数据加密，用户授权明确 |

### 7.2 关键假设

- 假设第三方大模型 API（如 OpenAI Whisper + GPT-4）能够准确将猫狗叫声映射到人类情绪标签
- 假设目标用户主要使用微信/Instagram/TikTok 作为分发渠道，手机端浏览器录音功能兼容性 > 90%
- 假设用户愿意注册账号以保存历史记录（约 60% 以上），少量游客也能使用基础分析

## 8. 里程碑与交付建议

### 8.1 版本规划

| 里程碑 | 时间（周） | 交付内容 |
|--------|------------|----------|
| MVP（最小可行产品） | 4 周 | 实现 P0 全部需求：用户注册、录音/上传、分析（猫狗、中英）、历史记录、TTS 播放。可用性测试 |
| V1.1（增强版） | 2 周（第 5–6 周） | 实现 P1 需求：结果详情页、分享功能、筛选历史、录音波形 |
| V1.2（优化版） | 2 周（第 7–8 周） | 实现 P2 部分需求：多宠物档案、离线缓存；性能优化；Bug 修复 |
| 正式上线 | 第 9 周 | TikTok 广告投放 + App Store/Google Play 网页版发布 |

### 8.2 交付建议

- **第 1–2 周**：完成 Next.js 项目脚手架、Supabase 数据模型、第三方 API 集成（语音识别 + 大模型 + TTS）、录音/上传 UI 开发
- **第 3 周**：联调核心分析流程（录制→识别→情绪→TTS→播放），修复边界问题
- **第 4 周**：内部测试 + 5 位真实用户试用，收集反馈并调整准确率基线
- **第 5–6 周**：根据反馈迭代，增加结果详情、分享功能
- **第 7–8 周**：性能压测、多语言完善、隐私合规文档上线
- **第 9 周**：TikTok 广告素材准备，监控系统（Sentry + Vercel Analytics）就绪，正式发布

### 8.3 资源预估

- **前端**：1 人（Next.js + Tailwind）
- **后端/集成**：1 人（Node.js API + Supabase + 第三方 API）
- **数据标注**：需收集至少 500 条猫狗叫声样本（公开数据集 + 早期用户贡献）用于模型调优（短期可先用通用模型）
- **第三方 API 预算**：语音识别（Whisper）约 $0.006/分钟；大模型推理（GPT-4o）约 $0.015/次；TTS（ElevenLabs）约 $0.0001/字符。预估每月 1000 次分析，成本约 $50–$100，可预充信用额度。
