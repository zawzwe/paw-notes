# 数据库设计文档

## 1. 数据库选型建议

- **主数据库**：Supabase（基于 PostgreSQL 15+），提供托管数据库、身份认证、存储、实时订阅功能，完美契合项目需求。
- **理由**：
  - 配合 Vercel 部署，后端使用 Node.js 的 Supabase SDK 直接操作。
  - 内置 Row Level Security (RLS) 满足基本合规需求。
  - PostgreSQL 支持 JSON、全文检索、索引优化，适合中等复杂度分析业务。
- **补充**：音频文件存储于 Supabase Storage（对象存储），数据库仅保存元数据与访问路径。

## 2. ER 关系概览

```
users 1 ── * recordings
recordings 1 ── 1 analyses
recordings * ── * audio_files (但实际存储为 Storage 对象，DB记录单文件路径)
```

- **users**：用户账户信息
- **recordings**：录音/上传记录（含请求来源、时间、状态）
- **analyses**：分析结果（识别含义、TTS 输出、置信度等）
- **audio_files**：音频文件元数据（可选，若直接存储在 recordings 表也可）

## 3. 核心表结构设计

### 3.1 users（用户表）

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | uuid | PRIMARY KEY, DEFAULT gen_random_uuid() | 用户唯一标识 |
| email | varchar(255) | UNIQUE, NOT NULL | 注册邮箱 |
| password_hash | varchar(255) | NOT NULL | bcrypt 哈希密码 |
| nickname | varchar(100) | | 用户昵称 |
| locale | varchar(10) | DEFAULT 'en' | 语言偏好 (en/zh) |
| created_at | timestamptz | DEFAULT now() | 注册时间 |
| updated_at | timestamptz | DEFAULT now() | 最后更新时间 |

### 3.2 recordings（录音/上传记录表）

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | uuid | PRIMARY KEY, DEFAULT gen_random_uuid() | 记录唯一ID |
| user_id | uuid | NOT NULL, FK → users.id | 所属用户 |
| source | varchar(20) | NOT NULL, CHECK (source IN ('realtime','upload')) | 来源：实时录音或文件上传 |
| species | varchar(10) | NOT NULL, CHECK (species IN ('cat','dog')) | 动物类型 |
| duration_sec | numeric(6,2) | | 音频时长（秒） |
| file_path | text | | 音频文件在 Storage 中的路径 |
| status | varchar(20) | DEFAULT 'pending', CHECK (status IN ('pending','processing','completed','failed')) | 分析状态 |
| created_at | timestamptz | DEFAULT now() | 录制时间 |
| updated_at | timestamptz | DEFAULT now() | 状态更新时间 |

### 3.3 analyses（分析结果表）

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | uuid | PRIMARY KEY, DEFAULT gen_random_uuid() | 结果唯一ID |
| recording_id | uuid | NOT NULL, UNIQUE, FK → recordings.id | 关联录音记录 |
| emotion_label | varchar(50) | | 情绪标签 (例如 happy, scared, hungry, upset) |
| emotion_confidence | numeric(4,3) | CHECK (emotion_confidence BETWEEN 0 AND 1) | 置信度 |
| translated_text | text | | 模型生成的“宠物人话”文本（英文） |
| translated_text_zh | text | | 中文翻译（可选，由模型输出或后端翻译） |
| tts_audio_path | text | | TTS 生成的音频文件 Storage 路径（MP3） |
| tts_language | varchar(5) | DEFAULT 'en' | TTS 输出语言 (en/zh) |
| raw_response | jsonb | | 大模型原始返回数据（调试用） |
| created_at | timestamptz | DEFAULT now() | 分析完成时间 |

### 3.4 audio_files（音频文件元数据表，可选）

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | uuid | PRIMARY KEY, DEFAULT gen_random_uuid() | 文件唯一ID |
| recording_id | uuid | NOT NULL, FK → recordings.id | 关联录音 |
| bucket | varchar(50) | NOT NULL | Supabase Storage bucket 名称 |
| file_path | text | NOT NULL | 完整路径（含文件扩展名） |
| file_size | bigint | | 文件大小（字节） |
| mime_type | varchar(100) | | 例如 audio/wav, audio/mp3, audio/webm |
| checksum | varchar(64) | | SHA256 哈希（可选） |

- **说明**：若 recordings.file_path 可直接代表主音频文件，此表可省略。为未来支持多版本/多格式保留。

## 4. 数据字典

- **emotion_label**：受控词典（cat/dog 通用）  
  - `happy`, `sad`, `angry`, `fear`, `excited`, `hungry`, `pain`, `playful`, `anxious`  
  - 由大模型返回，数据库不预置枚举，允许扩展。
- **status** 枚举含义：
  - `pending`：等待队列处理
  - `processing`：正在调用大模型 API 分析
  - `completed`：分析完成，TTS 已生成
  - `failed`：分析过程出错
- **source**：`realtime` 表示通过浏览器实时录音（MediaRecorder），`upload` 表示用户手动上传文件。

## 5. 索引与查询优化

### 5.1 主键索引（自动）
- users.id, recordings.id, analyses.id, audio_files.id

### 5.2 高频查询索引

| 索引名称 | 表 | 列 | 类型 | 用途 |
|----------|----|----|------|------|
| idx_recordings_user_id | recordings | user_id | B-tree | 用户查看自己的历史记录 |
| idx_recordings_status | recordings | status | B-tree | 后台轮询待处理任务 |
| idx_recordings_created_at | recordings | created_at DESC | B-tree | 按时间排序显示记录 |
| idx_analyses_recording_id | analyses | recording_id | UNIQUE, B-tree | 通过录音ID快速获取分析结果 |
| idx_users_email | users | email | UNIQUE, B-tree | 登录/注册查询 |

### 5.3 查询优化建议
- **历史记录分页**：使用 `created_at` + `id` 结合游标分页，避免 OFFSET 性能下降。
- **实时状态轮询**：前端通过 Supabase 实时订阅 `recordings` 表的 `status` 变更，减少无效轮询。
- **大模型调用**：采用异步队列（例如 Redis Bull 或 Supabase Edge Functions 的队列），避免阻塞 HTTP 响应。
- **JSON 查询**：`analyses.raw_response` 使用 jsonb 类型，可建立 GIN 索引若需内部查询，但暂时不必须。

## 6. 数据迁移与备份策略

### 6.1 数据迁移
- **环境**：开发、预发布、生产均使用 Supabase 项目，通过 Supabase CLI 管理。
- **迁移工具**：Prisma Migrate（配合 Supabase PostgreSQL），或直接使用 `pg_dump` + `pg_restore`。
- **流程**：
  1. 本地或 CI 运行 `prisma migrate dev` 生成迁移文件。
  2. 推送到生产时执行 `prisma migrate deploy`。
  3. 或使用 Supabase 的 **Database Branching** 功能进行预发布测试。

### 6.2 备份策略
- **自动备份**：Supabase 项目默认启用 **Point-in-Time Recovery (PITR)**，保留 7 天每日备份。
- **手动备份**：
  - 每周使用 `pg_dump -Fc` 将数据库全量导出至外部存储（如 AWS S3 或 Supabase 自身对象存储）。
  - 备份脚本通过 Vercel Cron Jobs 或 Supabase Edge Functions 定时触发。
- **对象存储备份**：Supabase Storage 中的音频文件建议开启跨区域复制（需手动配置），或定期拷贝至其他云（如 Backblaze B2）作为冷备。

### 6.3 灾难恢复
- 使用 PITR 恢复到任意时间点（7 天内）。
- 恢复步骤：
  1. 新建 Supabase 项目或使用数据库连接串覆盖。
  2. 通过 Supabase Dashboard 的“Restore”功能或 `pg_restore` 导入备份。
  3. 更新环境变量中的数据库连接。
