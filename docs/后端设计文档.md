# 后端设计文档

## 1. 技术栈推荐

| 层级 | 技术选型 | 版本说明 |
|------|----------|----------|
| 运行时 | Node.js (LTS) | 推荐 20.x 以上，支持 ESM |
| 框架 | Express / Fastify | Express 生态成熟，Fastify 性能更优 |
| 数据库 | Supabase (PostgreSQL) | 托管版，含 Auth、Storage、实时订阅 |
| 缓存 | Redis (Upstash) | 可选，用于音频文件临时缓存与限流 |
| 第三方API | OpenAI Whisper (语音识别) + Deepgram / Azure TTS | 按需调用，支持中英文 |
| 部署 | Vercel (Serverless Functions) | 自动扩缩，全球 CDN 边缘 |
| 监控 | Sentry + Logtail | 错误捕获与结构化日志 |

**关键说明**  
- 后端采用无服务器函数（Serverless Functions），每请求独立冷启动，需控制函数超时（Vercel 最大 60s，可配置）。  
- 音频文件上传由 Supabase Storage 处理，后端只获取公开/私有 URL 传递给 AI 服务。  
- Redis 用于存储高频访问的音频分析结果（TTL 10 分钟），减少重复调用第三方 API。

## 2. 系统架构概览

```
┌─────────────┐     ┌───────────────────┐     ┌───────────────┐
│   客户端    │────▶│  Vercel Edge      │────▶│  Supabase     │
│ (Next.js)   │     │  + Node.js API    │     │  (PostgreSQL) │
└─────────────┘     └────────┬──────────┘     │  + Auth       │
                             │                 │  + Storage    │
                             ▼                 └───────────────┘
                      ┌──────────────┐
                      │  第三方API    │
                      │  Whisper      │
                      │  TTS (Deepgram│
                      │  / Azure)     │
                      └──────────────┘
```

**核心组件**  
1. **API 网关**：Vercel Serverless Functions 接收请求，路由至各个处理函数。  
2. **音频处理引擎**：接受用户上传或实时录音的音频字节流，降噪后传给 Whisper 获取文本（并标记动物类型）。  
3. **语义分析层**：将 Whisper 输出的文本（如“Woof - high pitch”）加上上下文，调用 GPT-3.5/4 解析情绪并生成自然语言回复。  
4. **TTS 转录**：将生成的回复文本转为语音，输出 mp3 文件返回客户端。  
5. **持久化服务**：Supabase 存储用户信息、历史记录（音频路径、分析结果、TTS 路径）。  

**数据流方向**：客户端 → API → 文件暂存 → 第三方 AI → 结果缓存 → 存入 DB → 返回响应。

## 3. API 设计

### 3.1 认证模块 `/api/auth/*`  
| 方法 | 端点 | 描述 | 请求体 | 响应 |
|------|------|------|--------|------|
| POST | `/api/auth/signup` | 注册 | `{ email, password, name }` | `{ user, access_token }` |
| POST | `/api/auth/login` | 登录 | `{ email, password }` | `{ user, access_token }` |
| POST | `/api/auth/logout` | 登出 | – | `{ success: true }` |
| GET  | `/api/auth/me` | 获取当前用户 | header: `Authorization: Bearer <token>` | `{ id, email, avatar_url }` |

### 3.2 音频分析模块 `/api/analyze/*`  
| 方法 | 端点 | 描述 | 请求体/参数 | 响应 |
|------|------|------|-------------|------|
| POST | `/api/analyze/upload` | 上传音频文件 | multipart/form-data: `file` (audio/mpeg, wav, ogg) | `{ analysis_id, status: "processing" }` |
| POST | `/api/analyze/realtime` | 实时录音分析 | binary: `audio/webm` chunk | `{ partial_text, emotion, tts_url }` |
| GET  | `/api/analyze/status/:id` | 查询分析结果 | path: `analysis_id` | `{ id, original_audio_url, predicted_text, emotion, tts_url, createdAt }` |

**说明**  
- 实时录音分析采用 WebSocket 方案（Vercel 支持 WebSocket via `@vercel/functions`），建议使用 Server-Sent Events 回退。  
- 请求体中的音频文件首先保存至 Supabase Storage `audio-uploads` bucket，返回公开 URL 后再触发异步分析。  
- 分析结果保存至表 `analysis_history`，TTL 30 天后自动清理（Supabase 行级安全策略控制）。

### 3.3 用户历史记录 `/api/history/*`  
| 方法 | 端点 | 描述 | 查询参数 | 响应 |
|------|------|------|----------|------|
| GET  | `/api/history` | 列出当前用户历史 | `?page=1&limit=10&animal_type=cat` | `{ items: [...], total, next_cursor }` |
| DELETE | `/api/history/:id` | 删除单条记录 | – | `{ success: true }` |

## 4. 数据流与业务流程

### 4.1 上传音频 → 分析 → 结果返回
1. 用户通过 Next.js 前端选择音频文件或录制实时流。  
2. 前端将音频发送至 `/api/analyze/upload`（或 WebSocket）。  
3. 后端接收文件，转存至 Supabase Storage，获取公共 URL。  
4. 将 URL 和动物类型（前端选择 or 自动识别）传给第三方语音识别 API（Whisper）。  
5. Whisper 返回带时间戳的文本（如“汪——急促”）。  
6. 后端调用 GPT 模型：传入文本 + 动物类型 + 预设 prompt，输出情绪标签 + 拟人化句子（如“主人，我饿了！”）。  
7. 调用 TTS API 将句子转为语音，保存音频文件至 Storage。  
8. 将 `analysis_id` 及结果存入 Supabase `analysis_history` 表。  
9. 返回给客户端：JSON 包含情绪、TTS 预签名 URL。

**关键延迟点**：第三方 API 响应（约 2~5s），建议在返回前允许前端轮询 status 端点。

### 4.2 实时录音处理
1. 建立 WebSocket 连接（或 fallback SSE）。  
2. 客户端每 2 秒发送一个音频流 chunk。  
3. 后端累积 chunks 至 10 秒缓冲区，然后调用 Whisper（开启流式模式）。  
4. 每收到一次分析结果，更新 TTS 并推送回客户端。  
5. 用户停止录音后，一次性保存完整分析记录至数据库。

### 4.3 高并发应对
- **异步处理**：上传分析全部走消息队列（Redis Stream 或 Supabase Realtime + worker）。  
- **缓存热结果**：Redis 缓存最近 100 条分析结果，防重复调用。  
- **速率限制**：每个用户每分钟最多上传 5 次，使用 Upstash Redis 计数器实现。  
- **CDN 边缘**：静态资源（TTS 音频）通过 Vercel Edge Cache 缓存，降低源站压力。

## 5. 认证与权限方案

- **JWT 认证**：使用 Supabase Auth 内置能力，签发 Access Token（有效期 1 小时）和 Refresh Token（30 天）。  
- **角色模型**：仅普通用户（user）和管理员（admin，后台系统）。  
- **行级安全 (RLS)**：Supabase 表 `analysis_history` 启用 RLS，确保 `SELECT` 只有 `user_id = auth.uid()` 能查询自己记录。  
- **存储桶权限**：`audio-uploads` 和 `tts-audio` 存储桶默认为私有，前端通过预签名 URL（有效期 15 分钟）访问音频，防止越权。  
- **第三方 API Key 管理**：Whisper、GPT、TTS 的 API Key 存入环境变量（Vercel Environment Variables），函数运行时只在内网调用，不暴露给客户端。

## 6. 错误处理与日志策略

### 6.1 错误分类与响应格式
```json
{
  "error": {
    "code": "ANALYSIS_TIMEOUT",
    "message": "语音分析超时，请稍后重试",
    "details": "第三方 Whisper API 响应超过 15s"
  }
}
```
| HTTP 状态码 | 错误代码 | 说明 |
|-------------|----------|------|
| 400 | `INVALID_AUDIO` | 音频格式不支持 |
| 429 | `RATE_LIMIT_EXCEEDED` | 超出频率限制 |
| 502 | `EXTERNAL_API_FAILURE` | 第三方服务不可用 |
| 503 | `SERVICE_OVERLOAD` | 内部排队超时 |

### 6.2 日志策略
- 使用 `pino`（Node.js 高性能日志库）输出结构化 JSON 日志。  
- 关键日志点：请求入站、第三方 API 调用开始/完成、文件存储操作、最终结果返回。  
- 敏感信息过滤：自动遮蔽 Authorization 头和 audio 文件内容。  
- Sentry 集成：捕获未处理异常、慢请求（>5s）。  
- 日志保留期 7 天（Logtail 免费层），告警规则：错误率 > 5% 或 P95 延迟 > 10s 通知 Slack。

## 7. 部署与运维建议

| 项目 | 建议 | 理由 |
|------|------|------|
| CI/CD | GitHub Actions + Vercel Deploy | 自动部署，零停机 |
| 环境变量管理 | Vercel 环境变量分组（Production/Preview） | 隔离第三方 API Key |
| 数据库迁移 | Supabase CLI 管理 `migrations/` 文件夹 | 版本控制表结构变更 |
| 性能优化 | 开启 Supabase 连接池 + PGbouncer | 避免 Serverless 函数连接耗尽 |
| 成本控制 | 设置第三方 API 每月预算（如 $100/月），超额报警 | 防止恶意刷量 |
| 降级方案 | 当 Whisper/TTS 失败时，返回 `analysis_failed` 状态码并提示用户重试 | 保证可用性 |
| 监控大盘 | Datadog / Grafana（可选） | 跟踪 API 调用量、错误率、响应时间 |
| 备份策略 | Supabase 自动每日快照 + 手动导出 pg_dump | 数据安全（符合海外法规） |

**特殊说明**  
- Vercel 免费计划 Serverless 函数超时最大 60s，若分析过程耗时 > 50s 应改用**异步队列**（建议使用 Upstash Workflow 或 Supabase Cron + Webhook）。  
- 海外合规：不存储音频原始文件超过 30 天（设置 Supabase Storage 自动删除策略），用户可随时手动删除历史。
